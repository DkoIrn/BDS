Create two SQL migration files and a TypeScript types file.

**00003_datasets.sql:** Create the `datasets` table following the exact schema from the research:
- id (UUID PK, gen_random_uuid), job_id (FK to jobs ON DELETE CASCADE, NOT NULL), user_id (FK to auth.users ON DELETE CASCADE, NOT NULL)
- file_name (TEXT NOT NULL), file_size (BIGINT NOT NULL), mime_type (TEXT NOT NULL), storage_path (TEXT NOT NULL UNIQUE), status (TEXT NOT NULL DEFAULT 'uploaded')
- created_at (TIMESTAMPTZ DEFAULT NOW()), updated_at (TIMESTAMPTZ DEFAULT NOW())
- Indexes on job_id and user_id
- Enable RLS with policies for SELECT, INSERT, UPDATE, DELETE using auth.uid() = user_id
- Updated_at trigger using existing handle_updated_at() function

**00004_storage_bucket.sql:** Create the storage bucket and RLS policies:
- Insert into storage.buckets: id='datasets', name='datasets', public=false, file_size_limit=52428800 (50MB), allowed_mime_types array with 'text/csv', 'application/vnd.ms-excel', 'application/vnd.openxmlformats-officedocument.spreadsheetml.sheet'
- Storage RLS policies on storage.objects for INSERT (WITH CHECK), SELECT (USING), UPDATE (USING), DELETE (USING) -- all scoped to bucket_id='datasets' and folder matching auth.uid()::text using storage.foldername(name)[1]
- Use `TO authenticated` on all policies

**src/lib/types/files.ts:** Define TypeScript types:
- `DatasetStatus = 'uploaded'` (only successful uploads get DB records; more statuses added in later phases)
- `FileUploadStatus = 'queued' | 'uploading' | 'uploaded' | 'failed' | 'cancelled'` (client-side state machine for upload UI)
- `Dataset` interface matching the datasets table columns (id, job_id, user_id, file_name, file_size, mime_type, storage_path, status, created_at, updated_at)
- `FileUploadItem` interface for client-side queue state: { id: string, file: File, status: FileUploadStatus, progress: number, error?: string }
- Export constants: `ACCEPTED_FILE_TYPES` (the MIME type map for react-dropzone accept prop), `MAX_FILE_SIZE = 50 * 1024 * 1024`

**src/lib/actions/files.ts:** Create server actions following the established pattern (see projects.ts). All actions use 'use server', createClient from server, getUser() auth check, and return { error } or { success } objects.

1. `createFileRecord(data: { jobId: string, fileName: string, fileSize: number, mimeType: string, storagePath: string })` -- NOT a form action, just a plain async function called after client-side upload succeeds. Validates jobId exists and belongs to user (query jobs table with user_id filter). Inserts into datasets table with user_id from auth. Returns { success: true, id: string } or { error: string }. Calls revalidatePath for the job detail page.

2. `deleteFile(fileId: string)` -- Gets file record (RLS ensures ownership), deletes from storage.objects first (supabase.storage.from('datasets').remove([storagePath])), then deletes DB record. Returns { success: true } or { error: string }. Calls revalidatePath.

3. `getDownloadUrl(fileId: string)` -- Gets file record (verifies ownership via user_id check), generates signed URL with 300s (5 min) expiry using createSignedUrl. Returns { url: string } or { error: string }.

4. `getJobFiles(jobId: string)` -- Fetches all datasets for a job, ordered by created_at descending. Returns { data: Dataset[] } or { error: string }. Auth check + ownership verification through job ownership.
