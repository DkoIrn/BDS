    - Exact duplicate rows flagged as Critical

    - Unique rows produce no issues
    - KP values spaced beyond tolerance produce no issues

    Outliers (test_outliers.py):
    - Z-score: value beyond 3 sigma flagged as Warning with statistical context
    - Z-score: value within 3 sigma produces no issues
    - Z-score: column with zero std dev skipped (no crash, no issues)
    - Z-score: fewer than 3 data points skipped
    - IQR: value beyond Q1-1.5*IQR or Q3+1.5*IQR flagged as Warning
    - Explanation includes mean, std, z-score values

    Monotonicity (test_monotonicity.py):
    - Strictly increasing KP values produce no issues
    - Decreasing KP value flagged as Critical
    - Equal consecutive KP values flagged as Warning
    - Single row or empty data produce no issues

  <action>
    Implement each validator following RESEARCH.md code examples closely. Order of implementation:

    1. backend/app/validators/explanations.py -- format_location(kp_value, row_number) helper:
       - Returns "At KP {kp:.3f} (row {row})" when kp_value is not None
       - Returns "Row {row}" when kp_value is None
       - Used by all validators for consistent location formatting

    2. backend/app/validators/range_check.py -- check_range(df, column, min_val, max_val, tolerance, kp_column):
       - pd.to_numeric with errors="coerce"
       - effective_min = min_val - tolerance, effective_max = max_val + tolerance
       - Iterate out-of-range values, build ValidationIssue with Severity.CRITICAL
       - Message: "{location}: {COLUMN} value {actual}m exceeds {maximum|minimum} threshold of {limit}m (tolerance: +/-{tol})"

    3. backend/app/validators/missing_data.py:
       - check_missing_data(df, column, kp_column): detect isna() | empty strings, Severity.INFO
       - check_kp_gaps(df, kp_column, max_gap): sort KP, diff(), flag gaps > max_gap as Severity.WARNING
       - Dynamic gap threshold: if max_gap not provided, use median(diff) * 3

    4. backend/app/validators/duplicates.py:
       - check_duplicate_rows(df, kp_column): pandas duplicated(keep=False) for exact dupes, Severity.CRITICAL
       - check_near_duplicate_kp(df, kp_column, tolerance=0.001): sort KP, diff() < tolerance, Severity.CRITICAL

    5. backend/app/validators/outliers.py:
       - check_outliers_zscore(df, column, threshold=3.0, kp_column): scipy.stats.zscore, skip if std==0 or len<3, Severity.WARNING
       - check_outliers_iqr(df, column, multiplier=1.5, kp_column): Q1/Q3 via quantile(), flag beyond fences, Severity.WARNING
       - Include full statistical context in messages (z-score value, mean, std for z-score; Q1, Q3, IQR for IQR)

    6. backend/app/validators/monotonicity.py -- check_monotonicity(df, kp_column):
       - pd.to_numeric KP column, diff()
       - Decreasing (diff < 0) = Severity.CRITICAL
       - Equal (diff == 0) = Severity.WARNING
       - Message includes previous and current KP values

    7. backend/app/services/validation.py -- run_validation_pipeline(df, column_mappings, config):
       - Find KP column from mappings (first mapping with mappedType == "kp" and not ignored)
       - For each numeric mapped column (dob, doc, depth, top, elevation, easting, northing, latitude, longitude):
         - Run check_range if config has thresholds for that type
         - Run check_missing_data
         - Run check_outliers_zscore and check_outliers_iqr
       - Run check_kp_gaps if KP column exists
       - Run check_duplicate_rows
       - Run check_near_duplicate_kp if KP column exists
       - Run check_monotonicity if KP column exists
       - Return aggregated list of all ValidationIssue objects

       - Accept ValidateRequest body

       - Create validation_run record in Supabase

       - Update dataset status to "validated" (or "validation_error" on failure)
       - Return ValidateResponse
       - Use sync `def` (not async def) since supabase-py is synchronous (per RESEARCH.md Pitfall 1)
