  <name>Task 1: Server actions and severity utility</name>
  <files>src/lib/utils/severity.ts, src/lib/actions/validation.ts</files>
  <action>
1. Create `src/lib/utils/severity.ts` with shared severity helpers used across all dashboard components:
   - `getSeverityColor(severity: ValidationSeverity)` returns object with `badge`, `text`, `bg`, `border` Tailwind class strings. Critical = red/destructive, Warning = yellow, Info = blue.
   - `getSeverityIcon(severity: ValidationSeverity)` returns the Lucide icon component (AlertTriangle for critical, AlertCircle for warning, Info for info).
   - `getPassRateColor(rate: number)` returns border color class: green >= 90, yellow >= 70, red < 70.
   - `getVerdict(criticalCount: number)` returns `'PASS' | 'FAIL'` -- FAIL if any critical issues.
   - `formatRunDate(dateStr: string)` returns formatted date string (e.g., "Mar 14, 2:30 PM").

2. Add two new server actions to `src/lib/actions/validation.ts` (keep existing `getValidationRuns` and `getValidationIssues` untouched):

   - `getIssueContext(datasetId: string, rowNumber: number, contextSize: number = 5)`:
     Returns `{ headers: string[], rows: { rowNumber: number, cells: string[], isFlagged: boolean }[] }`.
     Implementation: Download file from Supabase Storage using dataset's storage_path, parse with PapaParse (CSV) or SheetJS (Excel) using existing patterns from parse API route, extract rows from `max(0, rowNumber - contextSize)` to `rowNumber + contextSize`, mark the flagged row with `isFlagged: true`. Handle ownership check via dataset.user_id. Return `{ error }` on failure.

   - `getJobValidationSummary(jobId: string)`:
     Returns `{ data: JobDatasetSummary[] }` where each entry has: `id, fileName, verdict ('PASS'|'FAIL'|null), issueCount, passRate, lastRunAt, isValidated`.
     Implementation: Fetch all datasets for the job with `select('id, file_name, status, validation_runs(id, total_issues, critical_count, pass_rate, run_at)')` ordered by run_at desc on the referenced table. For each dataset take the first (latest) run. Ownership check via user_id.

  Export the new `JobDatasetSummary` type from `src/lib/types/validation.ts` as well:
  ```typescript
  export interface JobDatasetSummary {
    id: string
    fileName: string
    verdict: 'PASS' | 'FAIL' | null
    issueCount: number
    passRate: number | null
    lastRunAt: string | null
    isValidated: boolean
  }
  ```

Create 5 components that compose into the per-dataset results dashboard. All are "use client" components.

1. **`results-stat-cards.tsx`** -- `ResultsStatCards` component:
   Props: `{ run: ValidationRun, onSeverityClick: (severity: ValidationSeverity | 'all') => void }`.
   - Prominent PASS/FAIL verdict banner at top using `getVerdict(run.critical_count)`. PASS = green border + CheckCircle icon. FAIL = red border + XCircle icon. Large bold text.
   - Three Card components in a responsive grid (3 cols on lg, 1 on mobile):
     - **Total Issues** card: Shows `run.total_issues` as large number. Below it, three clickable severity breakdown badges (critical/warning/info with counts). Clicking a badge calls `onSeverityClick` with the severity. Use `getSeverityColor` for badge styling.
     - **Pass Rate** card: Shows `run.pass_rate?.toFixed(1)%` as large number. Card has colored left border using `getPassRateColor(run.pass_rate)`. Show "N/A" if pass_rate is null.
     - **Data Completeness** card: Shows `run.completeness_score?.toFixed(1)%`. Show "N/A" if null.

2. **`issues-table.tsx`** -- `IssuesTable` component:
   Props: `{ issues: ValidationIssue[], datasetId: string, activeSeverity: ValidationSeverity | 'all', onSeverityChange: (severity: ValidationSeverity | 'all') => void }`.
   - Severity filter tabs at top using Tabs component (controlled via `activeSeverity`/`onSeverityChange`): All | Critical | Warning | Info. Each tab shows count derived from the issues array (NOT from run summary -- avoids mismatch per research pitfall #5).
   - Client-side sorting with useState for column (row_number, column_name, severity) and direction (asc/desc). Follow ProjectsTable sorting pattern exactly. Severity sort uses ordinal mapping: critical=0, warning=1, info=2.
   - Table with columns: Row # | Column | Severity | Message (truncated to ~80 chars).
   - Each row is clickable -- uses expandedRows Set<string> state. Clicking toggles expansion.
   - Use React Fragment pattern: main TableRow + conditional expanded TableRow with colSpan. Expanded row renders `<IssueRowDetail>`.
   - Cursor pointer + hover:bg-muted/50 on data rows.
   - Show "No issues found" empty state when filtered list is empty (with appropriate message per severity filter).

3. **`issue-row-detail.tsx`** -- `IssueRowDetail` component:
   Props: `{ issue: ValidationIssue, datasetId: string }`.
   - On mount, calls `getIssueContext(datasetId, issue.row_number)` to fetch surrounding rows.
   - Shows loading skeleton while fetching (use Skeleton component).
   - Displays:
     - Full explanation message (issue.message) with severity icon and color.
     - Expected vs Actual values in a two-column layout (if both present). Label: "Expected: {issue.expected}", "Actual: {issue.actual}".
     - Rule type badge showing issue.rule_type.
     - KP value if present: "KP: {issue.kp_value}".
     - Surrounding data context table: shows the 5+1+5 rows with headers. The flagged row is highlighted with a severity-colored background. Row numbers in first column.
   - Handle error state: show inline error message if getIssueContext fails. Don't break the table.

4. **`run-switcher.tsx`** -- `RunSwitcher` component:
   Props: `{ runs: ValidationRun[], selectedRunId: string, onRunChange: (runId: string) => void }`.
   - Uses shadcn Select component. Width ~280px.
   - Each SelectItem shows: formatted date + " -- {run.total_issues} issues".
   - Trigger shows selected run's formatted date.
   - Note: base-ui Select onValueChange returns string|null. Guard against null: `onValueChange={(v) => { if (v) onRunChange(v) }}`.

5. **`results-dashboard.tsx`** -- `ResultsDashboard` orchestrator component:
   Props: `{ datasetId: string }`.
   - On mount, fetches validation runs via `getValidationRuns(datasetId)`.
   - State: `runs` (ValidationRun[]), `selectedRunId` (string), `issues` (ValidationIssue[]), `loadingRuns` (boolean), `loadingIssues` (boolean), `activeSeverity` (filter state).
   - When selectedRunId changes, fetch issues via `getValidationIssues(selectedRunId)`. Clear issues and show loading skeleton during fetch (per research pitfall #3).
   - Latest run selected by default.
   - Layout:
     - Header row: "Results" heading on left. Run switcher on right (only if runs.length > 1). Verdict next to heading.
     - ResultsStatCards below header (pass onSeverityClick to set activeSeverity).
     - IssuesTable below stat cards.
   - Empty state: If no runs exist, show centered message "No validation runs yet. Run QC from the Mapping tab."
   - Loading state: Show skeleton cards while runs are loading.
